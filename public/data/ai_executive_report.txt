## 1. Executive Summary

During a targeted compromise assessment of our hybrid cloud infrastructure, CrowdStrike Falcon Identity Threat Detection (ITDR) flagged **three critical security incidents** of the highest severity (Risk Score: **100/100**). 

We have discovered a systemic, cross-platform offboarding failure resulting in "Zombie Administrators." Three employees who have been officially **Terminated** in our HR information system still possess **active, highly privileged sessions and administrative access keys** across Active Directory (AD), AWS IAM, and Okta. 

These identities represent an open back door. Because their sessions remain active, threat actors do not need to crack passwords or bypass Multi-Factor Authentication (MFA); they can simply hijack these orphaned, authenticated session tokens to execute a catastrophic, multi-cloud environment takeover.

### Identity 1: William Martin (Security) — EMP_1006

#### 1. Exact Blast Radius Risk
As a terminated Security Department administrator with active cross-platform privileges, this identity has the keys to our castle. A malicious actor exploiting these active sessions could:
*   **Disable Security Monitoring:** Terminate CrowdStrike Falcon sensors across the fleet, delete AWS CloudTrail logs, and disable GuardDuty detectors.
*   **Establish Persistence:** Create rogue IAM users, register backdoor SAML Identity Providers (IdPs) in Okta, and inject malicious Group Policy Objects (GPOs) in Active Directory.
*   **Neutralize Incident Response:** Lock out legitimate security personnel, delete backups, and blind the Security Operations Center (SOC) before executing a ransomware payload.

#### 2. Three-Step Urgent Remediation Plan

##### Step 1: Invalidate and Deactivate Okta Identity
Immediately terminate all active web and API sessions to stop active sessions.
*   **Action:** Log into the Okta Admin Console, navigate to **Directory > People > William Martin**, and click **Clear User Sessions**.
*   **API/CLI Execution:** 
    ```bash
    curl -X POST -H "Authorization: SSWS ${OKTA_API_TOKEN}" \
    "https://${ORG}.okta.com/api/v1/users/EMP_1006/lifecycle/deactivate"
    ```

##### Step 2: Sever Active Directory Access & Strip Memberships
Disable the AD account and strip all group memberships to prevent lateral movement to on-premises resources.
*   **Action:** Run PowerShell as Domain Admin to disable, move to a quarantined Organizational Unit (OU), and strip group memberships:
    ```powershell
    Disable-ADAccount -Identity "EMP_1006"
    Get-ADPrincipalGroupMembership -Identity "EMP_1006" | Where-Object { $_.Name -ne "Domain Users" } | Remove-ADPrincipalGroupMembership -Identity "EMP_1006" -Confirm:$false
    Move-ADObject -Identity (Get-ADUser "EMP_1006").DistinguishedName -TargetPath "OU=Terminated_Quarantine,DC=corporate,DC=local"
    ```

##### Step 3: Revoke AWS IAM Federated Sessions and Access Keys
Invalidate active STS (Security Token Service) assumed-role sessions and delete any persistent programmatic access keys.
*   **Action:** Apply an explicit inline deny policy to the IAM role/user and revoke active sessions using the AWS CLI:
    ```bash
    # Delete active IAM access keys
    aws iam delete-access-key --user-name william.martin@corporate.com --access-key-id <ACCESS_KEY_ID>
    
    # Put RevokeSplat policy to invalidate federated sessions older than the current time
    aws iam put-role-policy --role-name AWS_Security_Admin_Role --policy-name RevokeSessions --policy-document '{
      "Version": "2012-10-17",
      "Statement": [{
        "Effect": "Deny",
        "Action": "*",
        "Resource": "*",
        "Condition": {"DateLessThan": {"aws:TokenIssueTime": "'$(date -u +"%Y-%m-%dT%H:%M:%SZ")'"}}
      }]
    }'
    ```

*   **PII & Extortion:** Exfiltrate social security numbers, home addresses, compensation structures, and performance reviews to conduct targeted corporate espionage or double-extortion ransomware campaigns.
*   **Advanced Phishing:** Weaponize payroll/onboarding portals to launch spear-phishing campaigns against other executives.
*   **Downstream Hijacking:** Modify bank routing numbers in the HR system to divert payroll, or trigger rogue onboarding requests to provision shadow employees.

#### 2. Three-Step Urgent Remediation Plan

##### Step 1: Kill Okta Sessions & Revoke OAuth Tokens
Ensure the user is forcefully logged out of all HR applications (Workday, ADP, etc.) integrated via Okta.
*   **Action:** In Okta, navigate to the user's profile, click **More Actions**, select **Clear User Sessions**, and then select **Deactivate**.
*   **API/CLI Execution:**
    ```bash
    # Revoke all token grants linked to the user
    curl -X DELETE -H "Authorization: SSWS ${OKTA_API_TOKEN}" \
    "https://${ORG}.okta.com/api/v1/users/EMP_1026/clients"
    ```

##### Step 2: Disable Active Directory Account & Revoke Kerberos Tickets
Disable the account in AD and force a Kerberos ticket renewal (TGT) invalidation.
*   **Action:** Run PowerShell on a Domain Controller:
    ```powershell
    Disable-ADAccount -Identity "EMP_1026"
    # Set login hours to zero to block any attempts to re-authenticate
    Set-ADUser -Identity "EMP_1026" -Replace @{logonHours = New-Object byte[] 21}
    ```
*   *Note: To immediately invalidate current Kerberos tickets, force replication across all Domain Controllers:* `repadmin /syncall /Adep`.

##### Step 3: Revoke AWS Console and IAM Roles Access
Prevent access to AWS console sessions that may have been assumed via HR portal integration.
*   **Action:** Locate the federated role used by HR and apply an immediate Deny policy to the specific user's principal:

## 3. Root Cause & Strategic Recommendations

### Why Did This Happen?
This incident is the result of a **siloed Joiner-Mover-Leaver (JML) workflow**. When HR terminated these employees, the HR Information System (HRIS) did not dynamically trigger session revocation and account disabling across the downstream identity providers (Active Directory, Okta, and AWS). 

Legacy offboarding relies on human intervention or batch-processed scripts that fail to invalidate live, active web and API session tokens, creating a dangerous gap where terminated employees remain active in our digital environment.

```
       [ HRIS System (Workday/ADP) ] - User status set to "Terminated"
                    │
                    ▼ (Siloed Workflow / Manual Delay)
     [ Active Directory ] ─── [ Okta IdP ] ─── [ Cloud Providers (AWS) ]
            │                    │                     │
            ▼                    ▼                     ▼
     Account Enabled      Sessions Active       Access Keys Valid
```

### Strategic Remediation Actions

To prevent this vulnerability from recurring, we must implement three core architecture changes:

1.  **Deploy CrowdStrike Falcon Identity Threat Detection & Response (ITDR):** Integrate Falcon Identity with our Domain Controllers and Okta. This ensures any anomalous behavior from "dormant" or "terminated" accounts is automatically blocked in real time before it can spread laterally.
2.  **Automate API-Driven Offboarding Orchestration:** Build an automated webhook bridge between our HR system (e.g., Workday) and our Identity Providers (Okta/Active Directory). A termination event in HR must trigger an automated, zero-delay script that:
    *   Disables the AD account.
    *   Deactivates the Okta account.
    *   Calls APIs to invalidate all active web/OIDC sessions and AWS IAM credentials.
3.  **Enforce Continuous Access Evaluation (CAEP):** Configure Okta and AWS to use Continuous Access Evaluation. This ensures that any change in user status (such as termination or security policy change) immediately revokes active session tokens across all downstream SaaS and Cloud environments in near-real-time, rather than waiting for tokens to naturally expire.
